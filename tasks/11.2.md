## Design

### Overview
Add `KeyCode::BackTab` (Shift+Tab) handling to cycle `app.active_tab` backward through the 9 right-pane tabs, wrapping from tab 0 to tab 8.

### Current State
- `handle_task_list_input` in `src/tui/mod.rs` (line ~1493) has a `KeyCode::Tab` arm that increments `active_tab`: `(app.active_tab + 1) % 9`.
- There are 9 tabs (indices 0-8) defined in `tab_titles()` in `src/tui/tabs/mod.rs`.
- No `TAB_COUNT` constant exists; the magic number `9` is used inline.
- `KeyCode::BackTab` is not referenced anywhere in the codebase.

### Changes Required

#### 1. `src/tui/mod.rs` - `handle_task_list_input` function (~line 1495)
Add a `KeyCode::BackTab` arm immediately after the existing `KeyCode::Tab` arm:
```rust
KeyCode::BackTab => {
    app.active_tab = (app.active_tab + 8) % 9;
}
```
This uses modular arithmetic `(current + 8) % 9` which is equivalent to decrementing with wrap (avoids underflow on `usize`).

#### 2. `src/tui/mod.rs` - Tests
Add two tests following the existing `test_handle_input_tab_cycles` pattern:
- `test_backtab_cycles_tabs_backward`: Start at tab 2, press BackTab twice, assert tab goes 2 -> 1 -> 0.
- `test_backtab_wraps_from_first_to_last`: Start at tab 0, press BackTab, assert tab becomes 8.

Both tests use `key_event(KeyCode::BackTab, KeyModifiers::SHIFT)` and call `handle_input`.

### Design Decisions
- No `TAB_COUNT` constant is introduced since the existing code uses the literal `9` and refactoring is out of scope for this task.
- No changes needed to `handle_input` (the top-level dispatcher) since `BackTab` will flow through to `handle_task_list_input` via the existing default path.
- The questions-tab `Tab` handler (line ~1094) cycles between answer textareas, not tabs. BackTab support for that context is out of scope.
