## Description

Add a steering prompt textarea at the bottom of the Agent Activity tab (Tab 2) so the user can inject additional guidance to the agent while it is working.

Layout: split the Tab 2 inner area vertically into an activity scroll region (top, takes remaining height) and a steering textarea (bottom, fixed at 4 rows). The textarea should be hidden or collapsed to 0 rows when no agent is active for the selected task, and visible at all times otherwise so the user can pre-stage input.

Focus: `p` (while on Tab 2 and not already editing the prompt on Tab 0) focuses the textarea with a yellow border; Esc unfocuses. When focused, all keys are forwarded to the textarea. When unfocused, `p` is the only special key.

Submission: Enter (when not focused) or Ctrl+Enter (when focused) should send the textarea content as a mid-session prompt via the existing `send_prompt_async` path on the active opencode session for the selected task. On successful send the textarea is cleared. The sent text should also appear as a `[You]` banner in the activity log.

Agent consultation: the opencode session already streams responses continuously; no changes to the server side are needed. The user's steering input arrives as an additional prompt in the conversation, which the agent will naturally pick up at its next turn or when it checks in between tool calls.

## Design: Tab 2 Steering Prompt Textarea

### Overview

Add a `tui-textarea` input at the bottom of Tab 2 (Agent Activity) that lets the user inject mid-session guidance to an active agent. The design mirrors the existing Tab 0 supplemental prompt and Tab 4 comment draft patterns.

### Data Model Changes

**`Tab2State` (src/tui/tabs/agent_activity.rs)**

Add four new fields:

```rust
/// Steering prompt textarea for injecting guidance to the active agent.
pub steering_input: TextArea<'static>,
/// Whether the steering textarea currently has keyboard focus.
pub steering_focused: bool,
```

No new `AppMessage` variants are needed. The existing `SendPrompt { task_id, session_id, prompt }` message is reused for submission.

### Determining "Agent Active"

The task description says the textarea should be visible when an agent is active and hidden otherwise. The existing `thinking_tasks: HashMap<TaskId, (Instant, String)>` map in `Tab2State` already tracks exactly this -- it is populated on `set_awaiting_response` and cleared on `clear_thinking` (session completion or error). A new public method `is_agent_active(&self, task_id: &TaskId) -> bool` will check `self.thinking_tasks.contains_key(task_id)`. This is the correct signal because it spans the full duration of an agent session (not just the pre-streaming wait).

### Layout Changes

**`agent_activity::render()`**

Currently renders a single `Paragraph` into the full `area`. Change to:

1. Determine visibility: call `state.is_agent_active(task_id)` (the new helper).
2. If active: split `area` vertically using `Layout::default().direction(Vertical).constraints([Constraint::Min(0), Constraint::Length(6)])` -- the 6 rows give 4 rows of text plus 2 for the border. Render the existing activity paragraph into the top chunk and the `TextArea` widget into the bottom chunk.
3. If idle: use the full `area` for the activity paragraph (0 rows for textarea). The textarea is not rendered.

The textarea block title will be "Steering Prompt". When focused, the border is yellow (`Style::default().fg(Color::Yellow)`); when unfocused, default border.

The render signature changes from `render(frame, area, task_id, &Tab2State)` to `render(frame, area, task_id, &Tab2State)` -- no signature change needed since `Tab2State` now contains the textarea. However, `TextArea::widget()` requires `&self`, and `Tab2State` is passed as `&Tab2State`. Since `TextArea::widget()` takes `&self`, this is fine for rendering.

### Focus Management

**`tui::handle_input()` -- Tab 2 section**

The existing Tab 2 input handling block (lines 571-583 in `tui/mod.rs`) is extended:

```
if app.active_tab == 2 {
    if app.tab2_state.steering_focused {
        // Focused: all keys go to textarea except Esc and Ctrl+Enter
        if key.code == KeyCode::Esc {
            app.tab2_state.steering_focused = false;
            app.tab2_state.set_steering_unfocused_style();
            return None;
        }
        if key.code == KeyCode::Enter && key.modifiers == KeyModifiers::CONTROL {
            // Submit steering prompt (see Submission below)
            return submit_steering_prompt(app);
        }
        // Forward all other keys to the textarea
        app.tab2_state.steering_input.input(Input::from(key));
        return None;
    }
    // Unfocused mode
    match key.code {
        KeyCode::Char('p') if key.modifiers == KeyModifiers::NONE => {
            // Only focus if an agent is active for the selected task
            if let Some(task_id) = app.selected_task() {
                if app.tab2_state.is_agent_active(task_id) {
                    app.tab2_state.steering_focused = true;
                    app.tab2_state.set_steering_focused_style();
                }
            }
            return None;
        }
        KeyCode::Enter if key.modifiers == KeyModifiers::NONE => {
            // Submit when unfocused (if textarea has content)
            return submit_steering_prompt(app);
        }
        KeyCode::PageUp => { app.tab2_state.scroll_up(); return None; }
        KeyCode::PageDown => { app.tab2_state.scroll_down(); return None; }
        _ => {}
    }
}
```

Note: The `p` key conflict with Tab 0 is not an issue because Tab 0 and Tab 2 input handling are in separate `if app.active_tab == N` blocks. The task description says "while on Tab 2 and not already editing the prompt on Tab 0" -- since you can only be on one tab at a time, this is naturally satisfied.

### Submission Flow

A private helper function `submit_steering_prompt(app: &mut App) -> Option<AppMessage>` will:

1. Extract the textarea content via `app.tab2_state.steering_input.lines().join("\n")`.
2. If empty, return `None` (no-op).
3. Get the selected task's `task_id` and look up the active session via `app.workflow_engine.state(&task_id)` to get `session_id`.
4. If no active session (`session_id` is `None` or phase is not `Running`), return `None`.
5. Push a `[You]` banner into `tab2_state`: `app.tab2_state.push_banner(&task_id, format!("[You] {}", text))`.
6. Clear the textarea: `app.tab2_state.steering_input = TextArea::default()` (with default block style).
7. Unfocus: `app.tab2_state.steering_focused = false`.
8. Return `Some(AppMessage::SendPrompt { task_id, session_id, prompt: text })`.

The existing `SendPrompt` handler in `App::handle_message` (lines 690-738 of app.rs) will handle the async dispatch via `send_prompt_async`. No changes needed there.

### Trade-offs & Decisions

1. **Reuse `SendPrompt` vs. new message variant**: Reusing `SendPrompt` avoids adding a new `AppMessage` variant and leverages the existing async dispatch path. The only difference from the normal `SendPrompt` (which forwards answers) is that steering prompts don't pass through the workflow engine -- they go directly to the session. This is correct because steering input is supplemental guidance, not a structured agent interaction.

2. **`thinking_tasks` as activity signal**: This map already perfectly tracks the active-agent lifecycle. Using it avoids duplicating state or querying the workflow engine from the render path.

3. **Fixed 6-row allocation (4 text + 2 border)**: Matches the task spec of "fixed at 4 rows". The `Constraint::Length(6)` accounts for top and bottom borders.

4. **No agent parameter on steering prompts**: The existing `SendPrompt` handler looks up the agent from the `session_map`, so the correct agent is automatically used.

5. **Clear-on-submit**: On successful emission of `SendPrompt`, the textarea is cleared immediately (optimistic). If the send fails, the `SessionError` will be shown in the activity log but the text is already gone. This is acceptable because the `[You]` banner preserves what was sent.

#### Step 13: Add `submit_steering_prompt` helper function

Add a private helper function in `src/tui/mod.rs` (near the other helper functions like `apply_status_change`):

```rust
/// Attempts to submit the steering prompt textarea content as a mid-session prompt.
///
/// Extracts the text from `app.tab2_state.steering_input`, looks up the active
/// session from the workflow engine, pushes a `[You]` banner, clears the textarea,
/// and returns `Some(AppMessage::SendPrompt { ... })`. Returns `None` if the
/// textarea is empty, no task is selected, or no active session exists.
fn submit_steering_prompt(app: &mut App) -> Option<AppMessage> {
    let text: String = app.tab2_state.steering_input.lines().join("\n");
    if text.trim().is_empty() {
        return None;
    }
    let task_id = app.selected_task()?.clone();
    let wf_state = app.workflow_engine.state(&task_id)?;
    // Only allow submission when the workflow is in the Running phase with an active session.
    if wf_state.phase != crate::workflow::transitions::WorkflowPhase::Running {
        return None;
    }
    let session_id = wf_state.session_id.clone()?;
    // Push [You] banner.
    app.tab2_state.push_banner(&task_id, format!("[You] {}", text));
    // Clear and unfocus the textarea.
    app.tab2_state.reset_steering();
    Some(AppMessage::SendPrompt {
        task_id,
        session_id,
        prompt: text,
    })
}
```
